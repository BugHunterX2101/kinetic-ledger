 PRODUCT REQUIREMENTS DOCUMENT Finance Data Processing &amp; Access Control Backend Version: 1.0 | Status: Draft | Date: April 2026 A backend system for role-based financial record management, access control, and dashboard analytics. 1. Overview This document defines the requirements for a backend system that manages financial records, enforces role-based access control, and exposes aggregated analytics APIs. It is written to guide implementation of the assessment submission. Product Finance Data Processing &amp; Access Control Backend Scope Assessment submission — backend engineering evaluation Interface RESTful API (JSON) Auth JWT-based token authentication Database PostgreSQL (primary) — SQLite acceptable for local Primary Users Admins, Analysts, Viewers 2. Goals &amp; Non-Goals Goals Implement clean, role-enforced REST APIs for users and financial records Expose aggregated dashboard endpoints beyond basic CRUD Demonstrate proper validation, error handling, and status code usage Produce readable, maintainable code with clear separation of concerns Provide a clear README with setup, assumptions, and API reference Non-Goals Production deployment, high availability, or horizontal scaling Multi-currency support or real-time financial data integration Frontend UI or client-side rendering Multi-tenant or organisation-level data isolation 3. User Roles &amp; Permissions Three roles are defined. Role assignment is done by an Admin. All roles are mutually exclusive — a user holds exactly one role at a time. Role Definitions Viewer — read-only access to dashboard summary and record listing Analyst — all Viewer access plus record filtering, search, and trend analytics Admin — full system access including user management and record mutations Permission Matrix Action Viewer Analyst Admin View dashboard summary ✓ ✓ ✓ View financial records ✓ ✓ ✓ Filter / search records ✗ ✓ ✓ Create financial records ✗ ✗ ✓ Update financial records ✗ ✗ ✓ Delete / soft-delete records ✗ ✗ ✓ Manage users (create/update) ✗ ✗ ✓ Assign roles ✗ ✗ ✓ Access trend analytics ✗ ✓ ✓ 4. Functional Requirements 4.1 Authentication POST /auth/login accepts email + password, returns signed JWT JWT payload includes user ID and role; validated on every protected route Inactive users are rejected at login with 403 4.2 User Management Admins can create users (email, password, role) Admins can update role or active status of any user Admins can soft-deactivate users (is_active = false) All users can view their own profile 4.3 Financial Records Admins can create, update, and soft-delete records Viewers and above can list records (paginated, 20/page default) Analysts and above can filter by: date range, type, category Deleted records (is_deleted = true) are excluded from all listing queries 4.4 Dashboard &amp; Analytics /dashboard/summary — total income, total expenses, net balance /dashboard/by-category — per-category income and expense totals /dashboard/trends — monthly aggregates for the trailing 12 months /dashboard/recent — last 10 transactions, most recent first All dashboard queries exclude soft-deleted records 5. API Endpoint Reference Method Endpoint Role Required Description POST /auth/login Public Authenticate and receive JWT GET /users Admin List all users POST /users Admin Create a new user PATCH /users/:id Admin Update user or assign role DELETE /users/:id Admin Deactivate user (soft) GET /records Viewer+ List records with filters POST /records Admin Create a financial record PATCH /records/:id Admin Update a record DELETE /records/:id Admin Soft-delete a record GET /dashboard/summary Viewer+ Totals: income, expense, net GET /dashboard/by-category Analyst+ Breakdown by category GET /dashboard/trends Analyst+ Monthly/weekly trends GET /dashboard/recent Viewer+ Last N transactions All responses follow the envelope: { success, data, error, meta }. Pagination meta includes total, page, and per_page. 6. Data Model 6.1 Users Table Field Type Notes id UUID Primary key email String Unique, required password_hash String bcrypt hashed role Enum &apos;viewer&apos; | &apos;analyst&apos; | &apos;admin&apos; is_active Boolean Default true; false = deactivated created_at Timestamp Auto-set on insert 6.2 Financial Records Table Field Type Notes id UUID Primary key, auto-generated amount Decimal(12,2) Required, non-zero type Enum &apos;income&apos; | &apos;expense&apos; category String e.g. Salary, Rent, Travel date Date ISO 8601, defaults to today notes Text Optional description is_deleted Boolean Soft delete flag, default false created_by UUID (FK) References users.id created_at Timestamp Auto-set on insert updated_at Timestamp Auto-updated on change 7. Validation &amp; Error Handling Input validation is enforced at the route layer before any business logic executes. All errors return structured JSON with a message and optional field-level errors array. Scenario Expected Response Missing required fields 400 Bad Request — field-level error messages Invalid token / no token 401 Unauthorized Insufficient role 403 Forbidden Record not found 404 Not Found Duplicate email on register 409 Conflict Invalid date or amount 422 Unprocessable Entity Server / DB error 500 with generic error message 8.
